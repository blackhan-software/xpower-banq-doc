================================================================================
CONSISTENT EVENT PATTERN
================================================================================

Unlike settle transactions (which showed 9 logs vs 8 logs), redeem maintains a
consistent 6-log pattern across all transactions regardless of amount:

1. Transfer (sAPOW):          position tokens burned
2. Sync (sAPOW):              index updated
3. Transfer (vAPOW):          vault shares burned
4. Transfer (APOW):           underlying tokens to user
5. Redeem (Vault):            ERC4626 event
6. Redeem (Pool):             main pool event

This consistency occurs because redeem is simpler than settle:
  - No debt position to close
  - No interest distribution to suppliers
  - No large holder tracking updates
  - Single-direction flow: burn position -> receive tokens

================================================================================
COMPARISON WITH OTHER OPERATIONS
================================================================================

Operation | Cold Gas  | Warm Gas  | Savings | % Savings
----------|-----------|-----------|---------|----------
Borrow    | 278,858   | 249,199   | 29,659  | 10.64%
Settle    | 256,451   | 233,603   | 22,848  | 8.91%
Redeem    | 384,707   | 369,817   | 14,890  | 3.87%

Why Redeem Has Smallest % Savings:

1. Higher fixed costs: ERC4626 vault operations (preview, convert, redeem)
   add significant computation that doesn't benefit from storage warming

2. Two-layer abstraction: Redeem must interact with both Pool and Vault,
   doubling the token transfer overhead (SupplyPosition + Vault shares)

3. Larger base gas: At ~370-385k gas, redeem is the most expensive operation,
   so the ~15k warm savings represents a smaller percentage

4. No health check variability: Unlike borrow (which saves on oracle calls),
   redeem has consistent computation regardless of storage state

================================================================================
DUST REDEMPTION PATTERN
================================================================================

Similar to settle transactions, redemption requires multiple transactions:

  1st: 1,000.00 APOW    (bulk of supply)
  2nd:     0.000999 APOW (interest dust)
  3rd:   ~1 nanoAPOW    (final dust cleanup)

This occurs because:
1. Supply position balance changes with each index update
2. Vault share calculations have rounding
3. Interest accrues between transactions
4. Final dust must be claimed in a separate transaction

6. 3 transactions needed to fully close position due to rounding/interest
